Glossary: definitions of all terms and concepts

Every keyword and concept used in this handbook is defined here in full. Use this section when you encounter an unfamiliar term in the Get started, Install, Config, or technical sections.

Raspberry Pi

The Raspberry Pi is a single-board computer used as the compute platform for Smart Pi. We recommend a Pi 4 or Pi 5 for sufficient performance for OpenCV-based vision; a Pi Zero can run the app but will struggle (e.g. < 10 FPS, reduced resolution) with real-time detection and smoothing. The Pi runs Raspberry Pi OS (or compatible Linux), drives the projector over HDMI, and reads frames from a USB or CSI camera.

Projector and calibration

The projector is the display output: it shows the interactive content (e.g. Sandbox Draw, Particle Field) on a wall or table. Calibration is the one-time (or occasional) process that establishes a mapping from camera image coordinates to projector pixel coordinates. Without calibration, a pointer detected in the camera image cannot be drawn in the correct place on the projected image. Calibration is done by having the user tap four known corner targets; the software solves a homography from those four point correspondences and saves the result to configs/calibration.yaml.

Camera and pointer (IR pen, bright point)

The camera captures video of the projected surface. The pointer is the object the user moves to interact: either an IR pen (an infrared LED at the tip, visible to an IR-sensitive camera) or a bright visible spot (e.g. laser or LED). The software detects the pointer in each frame and produces a single (x, y) position in camera image space, which is then mapped to projector space via the calibration homography. IR mode is generally more robust under varying room light; bright-point mode is simpler to source (any webcam and laser pointer).

Bill of materials (BOM)

The bill of materials is the canonical list of parts needed to build the setup: Pi, projector, camera, cables, optional IR pen and mounts. The project provides hardware/bom.csv and the doc docs/01_bill_of_materials.md so that anyone can replicate the hardware without guessing. The BOM includes quantities, supplier notes, and references to sub-assemblies (e.g. IR pen parts in hardware/ir_pen/pen_bom.csv).

Configuration and profiles

Configuration is stored in YAML files under configs/. The main app config is configs/app.yaml: it specifies camera index, resolution, detector type (IR or bright), detection threshold, and a profile name. Profiles (e.g. configs/profiles/classroom.yaml, dark_room.yaml, daylight.yaml) override threshold and blur for different lighting. The app loads app.yaml and merges the chosen profile so that one logical config dict drives capture, detection, and display. Calibration output is written to configs/calibration.yaml (gitignored); its schema is documented in configs/calibration.yaml.example.

Capture, detect, track

Capture is the layer that reads frames from the camera (OpenCV or libcamera) at a target resolution and frame rate. Detect is the layer that, given one frame, finds the pointer: either the brightest IR blob (contour-based) or the brightest pixel (for visible bright point). It outputs one (x, y) in camera coordinates or none. Track is the layer that smooths the (x, y) stream over time using an exponential moving average (EMA) and optional velocity clamping so the cursor does not jitter or jump when the pointer is briefly lost.

Homography and calibration.yaml

A homography is a 3×3 matrix that maps points in one plane (the camera image) to points in another plane (the projector image). Smart Pi uses a 4-point homography: the user taps four corners in a fixed order, the app records the four camera-space points, and OpenCV’s findHomography computes the matrix. The matrix and the projector resolution at calibration time are saved to configs/calibration.yaml. At runtime, every detected camera (x, y) is transformed with this matrix to get projector (x, y) for drawing and events.

PointerEvent and gestures

A PointerEvent is the normalized event emitted by the interaction layer: it has (x, y) in projector space, a state (down, move, or up), a timestamp, and optional confidence. The gesture layer turns the stream of positions (or “pointer lost”) into a stream of PointerEvents: pointer appears → down; pointer moves → move; pointer disappears → up. Higher-level gestures (tap, drag, long-press) are derived from short sequences of these events (e.g. down then up within a time and distance threshold is a tap). Adapters (Pygame, WebSocket, uinput) consume PointerEvents and deliver them to scenes or external clients.

Renderer and scenes

The renderer is the main application loop: it owns the camera capture, runs the detector and tracker, applies the homography, emits pointer events, and draws the active scene plus an optional overlay (FPS, threshold, pointer position). Scenes are the interactive content: Sandbox Draw (draw with the pointer, corner hotzone to change color), Particle Field (pointer pushes particles), and STEM Widgets (ruler, angles, graph paper). The scene receives pointer events as dicts and implements on_pointer and draw (and optionally update for animation).

CLI and scripts

The CLI is the command-line entry point: smartpi run, smartpi calibrate, smartpi bom, smartpi healthcheck, smartpi export-diagnostics. Scripts in scripts/ wrap these: install_pi.sh installs Python dependencies, healthcheck.sh runs display and camera checks, calibrate.sh and run_demo.sh delegate to the CLI. Services (systemd unit, autostart .desktop) allow the app to start on boot for kiosk use.

Healthcheck and diagnostics

Healthcheck verifies that the camera is detectable and reports resolution and FPS (and optionally CPU temperature on the Pi). It uses the same config as the app so that the same camera and resolution are tested. Diagnostics export bundles system info and config (excluding calibration and secrets) into a tarball for bug reports; it is implemented by smartpi export-diagnostics and scripts/export_diagnostics.sh.

Homography matrix

The homography matrix is a 3×3 matrix of real numbers that maps two-dimensional points in the camera image plane to two-dimensional points in the projector image plane. OpenCV’s findHomography computes it from four point correspondences (four camera points and four projector points). At runtime, each detected (x, y) in camera space is converted to homogeneous coordinates, multiplied by the matrix, and then dehomogenized to get (px, py) in projector space. The matrix is stored in configs/calibration.yaml in row-major order.

EMA (exponential moving average)

Exponential moving average is the smoothing method used by the tracker. The tracker maintains a current position; when a new detection arrives, the position is updated as a weighted blend of the old position and the new point (e.g. new = alpha * detected + (1 - alpha) * old). This reduces jitter when the detector’s raw output varies frame-to-frame. When the pointer is lost (no detection), the tracker can hold the last position for a short time or reset; the exact behavior is defined in the vision tracking module.

BGR and camera frame

BGR is the pixel format used by OpenCV for color images: each pixel has three channels in Blue-Green-Red order. The camera capture layer returns frames in BGR (or grayscale if configured). The detection layer typically converts the frame to grayscale before thresholding; for IR or bright-point detection, a single channel is sufficient. The projector output is driven by the renderer (e.g. Pygame surface), which uses its own color representation; the pipeline does not store BGR beyond the capture and detect steps.

Contour and blob

In image processing, a contour is the boundary of a connected region of pixels (e.g. after thresholding). A blob is a contiguous region; in Smart Pi the IR detector finds blobs by thresholding the image, optionally applying morphological operations, and then finding contours. The “brightest” or largest blob is chosen as the pointer; its centroid (or peak) gives the (x, y) in camera coordinates. The bright-point detector may use a simpler method (e.g. global max pixel) instead of contours.

Adapter (Pygame, WebSocket, uinput)

An adapter is the component that receives the stream of PointerEvents from the interaction layer and delivers them to a consumer. The Pygame adapter injects events into the Pygame event queue or calls the scene’s on_pointer directly so that the built-in scenes (Sandbox Draw, Particle Field) receive (x, y) and state. The WebSocket adapter serializes events and sends them over a socket so that a web front-end or another process can drive visuals. The uinput adapter (Linux) creates a virtual mouse or touch device and emits kernel input events so that the pointer controls the system cursor or touch events. Swapping adapters does not require changing scene logic.

Scene (Sandbox Draw, Particle Field, STEM Widgets)

A scene is one interactive application shown by the renderer. Sandbox Draw lets the user draw with the pointer and change colors via a corner hotzone. Particle Field simulates particles that are pushed by the pointer. STEM Widgets provide ruler, angles, and graph-paper tools. Each scene implements the same interface: it receives pointer events (down, move, up) and optionally time-step updates, and it draws to the renderer’s surface. The renderer runs one scene at a time; scene selection is part of the run flow (e.g. menu or CLI argument).

Calibration wizard

The calibration wizard is the user-facing flow that produces configs/calibration.yaml. It projects four corner targets on the surface; the user taps each target with the IR pen or holds the pointer steady. The app records the pointer position in camera space for each corner, then solves the homography from the four point correspondences and saves the result. Optionally it runs a validation step (e.g. crosshair grid and reprojection error check). The wizard is designed to be completable in under two minutes with large “Next” buttons and minimal typing.

Reprojection error

Reprojection error measures how well the calibration fits the data: for a known point in projector space, we transform it back to camera space using the inverse homography, then compare to the actual camera-space detection. Alternatively we project camera points to projector space and compare to the expected projector positions. A high median error (e.g. above a threshold like 15 px at 1080p) indicates poor alignment; the app can warn the user to re-run calibration or adjust camera position. The exact formula and threshold are documented in the vision calibration and troubleshooting docs.

Get started

Smart Pi is designed so that a new user can go from a fresh Raspberry Pi and hardware to a running interactive demo by following a single path: prepare the Pi, install system dependencies and the project, connect the projector and camera, run a healthcheck, run the calibration wizard once, and then run the demo. This section explains that path at a high level; the Install section gives the exact commands.

You need a Raspberry Pi 4 or 5 (recommended), an HDMI projector, and a camera (USB or CSI). For the pointer you can use either an IR pen with an IR-sensitive camera or a laser or bright LED with a standard webcam. The project documentation and Bill of Materials describe both options and list parts. After you clone the repository and run the install script (or pip install -r requirements.txt and optionally pip install -e . for development), you connect the hardware and set the display resolution. Running the healthcheck script confirms that the camera is detected and reports approximate FPS. The calibration wizard then guides you through tapping four corners so the software can compute the camera-to-projector mapping; that mapping is saved and used every time you run the app. Finally, smartpi run (or ./scripts/run_demo.sh) starts the interactive app: you choose a scene (e.g. Sandbox Draw or Particle Field) and interact with the pointer. No bullet list can replace the two paragraphs above: the get-started flow is a linear sequence of steps (prepare, install, connect, healthcheck, calibrate, run), each step depending on the previous one, with the goal of a working interactive projection in under an hour for a first-time user.

Install

Installation has two parts: system dependencies (packages for Python, OpenCV, and Pygame) and project dependencies (the Smart Pi Python package and its pip requirements). On Raspberry Pi OS you install system packages with apt and project packages with pip. The build guide in the repository gives the exact commands; this handbook summarizes them and explains what each step provides.

System dependencies

First, prepare the Pi: flash Raspberry Pi OS to a microSD card, boot, and run sudo apt update && sudo apt upgrade. Optionally increase GPU memory via raspi-config for display.

Install system packages: sudo apt install -y python3-pip python3-venv libopencv-dev python3-opencv libsdl2-dev libsdl2-image-dev libportmidi-dev. This provides Python 3, OpenCV (for camera and image processing), and the SDL libraries required by Pygame. If you use a CSI camera with libcamera, also install libcamera-dev python3-libcamera.

Project install

Clone the repo and install the project: git clone https://github.com/YOUR_ORG/rashbarrypi.git, cd rashbarrypi, then ./scripts/install_pi.sh (which runs pip install -r requirements.txt) or manually pip install -r requirements.txt. For development you can run pip install -e . so that changes to the source are reflected without reinstalling. The install step does not create a virtual environment by default; you can create one with python3 -m venv .venv and activate it before running pip. After installation, connecting the projector and camera and disabling screen blanking completes the setup; the next step is healthcheck and calibration.

Config options

All configuration is under the configs/ directory. The main file is configs/app.yaml: it defines the camera device index, the capture resolution (width and height), the detector type (IR or bright), the detection threshold (0–255), and the name of the lighting profile to use.

App config and profiles

The profile name (e.g. classroom, dark_room, daylight) selects a file under configs/profiles/; that file’s keys are merged into the app config so that threshold, blur, and optional exposure hints can vary by environment. This design keeps a single place (app.yaml) for the camera and detector choice while allowing lighting-specific tuning without editing app.yaml.

Calibration and logging

configs/calibration.yaml is produced by the calibration wizard and must not be committed to version control. It contains the homography matrix (nine floats in row-major order), optional surface bounds, and the projector resolution at calibration time. The schema is documented in configs/calibration.yaml.example. configs/logging.yaml is optional: if present, the application configures Python logging from it (level and optional log file path). The project does not expose network ports by default; if a WebSocket or HTTP server is added later, ports and URLs will be documented in the config reference and README. No bullet list is used here without explanation: the three main config concepts are (1) app plus profile for runtime behavior, (2) calibration as generated output for the homography, and (3) logging as optional observability, each serving a distinct role in the system.

Project outcomes

Primary outcomes

Smart Pi aims to deliver a replicable, open source interactive projection system that anyone can build with a Pi, a projector, and a camera. The primary outcome is a working demo: after following the build guide and running calibration once, the user can draw with an IR pen or laser pointer on the projected image and interact with at least two scenes (Sandbox Draw and Particle Field). A secondary outcome is a clear codebase and documentation set so that contributors can add new scenes, detectors, or renderers without rewriting the core pipeline. The roadmap (v0.1 through v1.0) describes phased outcomes: hackathon demo, workshop-ready reliability, multi-pointer support, and finally a community kit with hardware compatibility data and repair-forward guides.

Community

Outcomes for the community include a single vocabulary (this handbook and the glossary), a full technical documentation set (the docs folder and technical deep dives that describe every function’s inputs and outputs), and a contribution process (CONTRIBUTING.md, issue templates, CODEOWNERS) so that collaborators know how to report bugs, propose features, and submit changes. The project explicitly states scope: in scope are Raspberry Pi, Linux, camera-based pointer tracking, homography calibration, and the provided demo scenes; out of scope are Windows or macOS host support, official pre-built hardware kits, and commercial support. Defining outcomes and scope in prose (with two paragraphs here) ensures that implementers and contributors align on what “done” means and what the project will and will not do.

Definition of done for v0.1: Power on Pi and projector → app can autostart → calibrate in under 2 minutes → draw and interact with particles; repository includes BOM, build guide, and documentation for adding a new scene.

Pipeline: services and data flow

Stage order

The Smart Pi pipeline is an ordered sequence of stages: configuration loading, capture, detect, track, calibrate (apply homography), interact (emit pointer events), render (draw scenes and overlay), and operate (CLI and scripts). Each stage consumes the output of the previous stage. Configuration is loaded once at startup and provides camera index, resolution, detector type, threshold, and profile-derived options. Capture reads one frame at a time from the camera and outputs a BGR image (or a failure flag). Detect takes that frame and returns either one (x, y) in camera coordinates or none. Track takes the stream of (x, y) or none and outputs a smoothed (x, y) or none, with internal state reset when the pointer is lost. The homography (loaded from calibration) maps camera (x, y) to projector (x, y). The interaction layer turns the stream of “current pointer or none” plus “previous pointer and down state” into a stream of PointerEvents (down, move, up). Adapters convert those events into the format expected by the renderer (e.g. a dict for Pygame scenes). The renderer draws the active scene and overlay and flips the display; the operate layer is the CLI and scripts that start the app, run calibration, or run healthcheck.

Data flow

This order is fixed so that data flows in one direction and each component has a single responsibility. The pipeline is documented in the repository in docs/04_system_architecture.md and in the technical deep dives (docs/technical/), where every function’s inputs and outputs are specified. Implementers and contributors should read the pipeline description and the glossary entries for Capture, Detect, Track, Homography, PointerEvent, and Renderer so that new code fits the existing contracts. No bullet list stands alone: the two paragraphs above explain the full order and the role of each stage so that the reader can trace any datum from hardware input to displayed output.

Functions and logic: technical reference

This section defines every public contract (function and method signature, data in, data out) and the logic for each part of the system in pipeline order. It satisfies best-practice technical documentation: implementers can see exactly what each function accepts, returns, and does without opening the source. The repository’s docs/technical/ deep dives repeat and expand this in markdown; the handbook is the single-page reference.

Modules and pipeline order

The codebase is organized in pipeline order: config (no stage; prerequisite) → capture → detect → track → calibrate (apply) → interact → render → operate. Config loading is shared by CLI and app; capture is the only source of frames; detect produces one camera-space point or None; track smooths in projector space; calibration maps camera to projector; interact turns position stream into PointerEvents; render runs the main loop and scenes; operate is the CLI (run, calibrate, bom, healthcheck, export-diagnostics).

Contracts and logic by layer

Config

_load_app_config(config_dir: str) → dict. Data in: config_dir (path to config directory). Data out: single merged dict with camera_index (int), resolution [width, height], detector ("ir" | "bright"), threshold (int), blur (int), and any keys from the selected profile. Logic: load app.yaml; read profile (default "classroom"); load profiles/<profile>.yaml if present and merge (profile overrides app); return merged dict. Calibration is not loaded here.

Capture (core/capture.py)

CameraCapture.__init__(camera_index, width, height, fps). Data in: four parameters (int, int, int, float). Data out: none; self._cap = None. Logic: store parameters; no device opened until open().

open() → bool. Data in: none (uses stored params). Data out: True if device opened and properties set, False otherwise. Logic: cv2.VideoCapture(camera_index); if not opened return False; set CAP_PROP_FRAME_WIDTH, HEIGHT, FPS; return True.

read() → tuple[bool, np.ndarray | None]. Data in: none. Data out: (success, frame); frame is BGR shape (H,W,3) or None on failure. Logic: if _cap None or not opened return (False, None); ret, frame = self._cap.read(); if not ret or frame None return (False, None); return (True, frame).

release(). Data in: none. Data out: none. Logic: if _cap set, call release() and set _cap to None.

Optional: create_capture(camera_index, width, height, fps, use_libcamera=True) returns LibcameraCapture (Picamera2) when available; otherwise CameraCapture. Same interface: open, read, release.

Preprocess and detect (vision)

preprocess_grayscale(frame, blur_ksize) → np.ndarray. Data in: BGR or gray frame; blur_ksize (int, odd or 0). Data out: grayscale (H,W) uint8. Logic: BGR→gray if 3 channels; Gaussian blur if blur_ksize > 0 and odd; return.

threshold_mask(gray, threshold) → np.ndarray. Data in: gray (H,W), threshold (int 0–255). Data out: binary mask 0/255. Logic: cv2.threshold(..., THRESH_BINARY).

detect_ir_blob(frame, threshold, blur_ksize, morph_ksize) → tuple[float, float] | None. Data in: frame, threshold, blur_ksize, morph_ksize. Data out: (cx, cy) centroid in camera space or None. Logic: grayscale+blur → threshold_mask → morphology (close/open) → findContours → largest contour by area → area ≥ 10 → moments → (cx, cy).

detect_bright_point(frame, threshold, blur_ksize) → tuple[float, float] | None. Data in: frame, threshold, blur_ksize. Data out: (x, y) of brightest pixel or None if max < threshold. Logic: preprocess_grayscale → minMaxLoc → if max_val ≥ threshold return max_loc else None.

Track and calibration (vision)

SmoothingTracker.__init__(alpha=0.3, max_velocity=500.0, use_kalman=False, process_noise=1e-2, measurement_noise=10.0). Data in: alpha, max_velocity (EMA path); use_kalman, process_noise, measurement_noise (Kalman path). Data out: none. Logic: store params; _x, _y = None. When use_kalman True, a 2D constant-velocity Kalman filter is used instead of EMA.

update(x, y) → tuple[float, float]. Data in: x, y (float, projector space). Data out: smoothed (x, y). Logic: if use_kalman then _update_kalman else EMA path (first sample, clamp, EMA); return (_x, _y).

reset(). Data in: none. Data out: none. Logic: _x = _y = None. Call when pointer lost.

solve_homography(src_pts, dst_pts) → np.ndarray | None. Data in: list of ≥4 (x,y) camera points, list of same length projector points. Data out: 3×3 float64 homography or None. Logic: if len < 4 return None; cv2.findHomography(src, dst, RANSAC, 5.0); return H.

save_calibration(path, homography, bounds, resolution). Data in: path (str|Path), H (3×3), bounds (x0,y0,x1,y1), resolution (w,h). Data out: none. Logic: serialize to YAML and write.

load_calibration(path) → dict | None. Data in: path. Data out: dict with homography, bounds, resolution or None. Logic: if path missing return None; load YAML; if no "homography" return None; return dict.

apply_homography(H, x, y) → tuple[float, float]. Data in: H (3×3), x, y (float, camera). Data out: (px, py) projector pixels. Logic: cv2.perspectiveTransform single point.

reprojection_error(calibration_path, src_pts, dst_pts) → float | None. Data in: path, camera points, projector points. Data out: median error in pixels or None. Logic: load calibration; for each pair apply_homography and compute distance; return median.

Interaction (events, gestures, adapters)

PointerEvent: dataclass x, y (float), state (PointerState), timestamp (float), confidence (float). PointerState: enum DOWN, MOVE, UP.

emit_events_from_track(xy, prev_xy, prev_down) → Iterator[PointerEvent]. Data in: xy (x,y)|None, prev_xy (x,y)|None, prev_down (bool). Data out: zero or more PointerEvent. Logic: if xy None and prev_down yield UP at prev_xy; if xy not None and prev_xy None and not prev_down yield DOWN at xy; if xy and prev_xy and position changed yield MOVE at xy.

classify_gesture(events) → str. Data in: list of PointerEvent. Data out: "tap" | "drag" | "long_press" | "none". Logic: require DOWN and UP; dt and distance; if dt ≥ 0.5s "long_press"; if dist>20 or moves>2 "drag"; else "tap".

pygame_event_from_pointer(ev) → dict. Data in: PointerEvent. Data out: dict with type "pointer", x, y, state (str), timestamp. Logic: build dict for scene on_pointer.

push_to_pygame(ev, callback). Data in: PointerEvent, callable. Data out: none. Logic: callback(pygame_event_from_pointer(ev)).

WebSocket adapter: send_pointer_event(ev, socket) serializes the PointerEvent to JSON and sends it over the given socket (send/sendall). Uinput adapter: emit_mouse(ev, uinput_fd) emits Linux mouse events via evdev when available (pass None for auto-created device; uinput_fd is reserved for future use).

Renderer and UI

run_app(fullscreen, config_dir). Data in: fullscreen (bool), config_dir (str). Data out: none; does not return until quit. Logic: _load_config; load_calibration (or projector_profile for display size when no calibration); Pygame init; create_capture(..., config.use_libcamera).open(); FPSCounter, SmoothingTracker(..., use_kalman=config.get('use_kalman')); scenes dict; main loop: read frame → detect → camera_to_proj → tracker.update/reset → emit_events_from_track → scene.on_pointer per event → prev_xy/prev_down update → scene.update/draw → draw_overlay → flip; finally release, pygame.quit.

_load_config(config_dir) → dict. Data in: config_dir. Data out: app dict. Logic: load app.yaml (and in correct implementation merge profile like _load_app_config).

Scenes: on_pointer(ev: dict), update(), draw(surface). Data in: ev has x, y, state; surface is Pygame Surface. Data out: none; state and pixels updated. SandboxDraw: corner hotzone cycles color; down/move/up build strokes. ParticleField: pointer pushes particles; update() steps physics. StemWidgets: ruler (length), angles (degrees), graph paper (grid + snap); mode and clear via corners.

draw_overlay(surface, fps, pointer_xy, threshold). Data in: surface, fps (float), pointer_xy (x,y)|None, threshold (int). Data out: none. Logic: draw FPS, threshold, pointer text and green circle at pointer.

run_wizard(config_dir, use_pygame=True). Data in: config_dir. Data out: none; writes calibration.yaml. Logic: load config; open capture; state machine target→solving→validate→done; collect 4 camera points; solve_homography; save_calibration; optional reprojection_error for validation.

CLI and operations

main(): Click group; subcommands run, calibrate, bom, healthcheck, export-diagnostics.

run(fullscreen, config_dir). Data in: from flags. Data out: none. Logic: import run_app; run_app(fullscreen, config_dir).

calibrate(config_dir). Data in: config_dir. Data out: none. Logic: import run_wizard; run_wizard(config_dir).

bom(repo, pen). Data in: repo path (--repo, default "."), pen flag (--pen). Data out: prints main BOM (and optionally pen BOM) to stdout; exit 1 if main BOM file missing.

healthcheck(config_dir). Data in: config_dir. Data out: prints PASS/FAIL, resolution, FPS, optional CPU temp. Logic: _load_app_config; open VideoCapture; read 30 frames with FPSCounter; release; echo; read thermal zone if present.

export_diagnostics(output, config_dir, no_config). Data in: output path, config_dir, no_config. Data out: prints path. Logic: bundle_diagnostics(output, include_config=not no_config, config_dir); echo path.

get_system_info() → dict[str, str]. Data in: none. Data out: platform, python, processor. Logic: platform module.

bundle_diagnostics(output_path, include_config, config_dir) → Path. Data in: output path, include_config (bool), config_dir. Data out: resolved Path to .tar.gz. Logic: create tarball; add system_info.txt (timestamp, platform, python); if include_config add app.yaml and profiles (exclude calibration.yaml); return path.

For step-by-step logic, edge cases, and failure modes, see docs/technical/ (01_system_data_flow through 07_cli_config_and_operations).

Contributing and open source development

How to contribute

Smart Pi is developed as an open source project so that the community can implement, extend, and replicate it. The repository includes a CONTRIBUTING.md file that describes the types of contributions (bug reports, feature ideas, hardware compatibility reports, code, documentation, and hardware designs), how to run the project locally, how to add sample data for tests, code style (Ruff, Black, type hints), and the pull request process. The code of conduct is linked from the README and CONTRIBUTING so that all participants share the same standards. Issue templates (bug report, feature request, hardware compatibility) guide reporters to provide enough context; the pull request template asks for a related issue, description, testing done, and a checklist (e.g. docs updated).

Process

To maximize collaboration, the project maintains a single vocabulary (this handbook and the glossary), a full documentation set (overview, build guide, architecture, calibration UX, interaction design, troubleshooting, safety, accessibility, config reference, API overview, and technical deep dives), and a clear scope and roadmap. Contributors can start with “good first issue” labels and small tasks; maintainers are listed in MAINTAINERS.md and CODEOWNERS. The project does not make private design decisions that affect contributors; discussion happens in issues or in Architecture Decision Records (docs/adr/). By providing get started, install, config, outcomes, pipeline, and function-level documentation in one place (and in the repo docs), the handbook supports both users who want to run the demo and developers who want to implement or extend the system.
