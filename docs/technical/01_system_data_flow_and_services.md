# System Data Flow and Services

This document describes the complete data flow and order of services in the Smart Pi Interactive Projection system. It covers every stage from hardware input (camera) to output (projected display and events), including data types, function call order, and the logic that connects each layer.

## Purpose and scope

Smart Pi turns a camera view of a projected surface into a stream of pointer events (tap, drag, long-press) in projector coordinates. The pipeline is strictly ordered: **Capture → Detect → Track → Calibrate (one-time) → Interact → Render → Operate**. Each stage consumes the output of the previous stage and produces typed data for the next. Configuration (YAML) and CLI commands control which components run and with what parameters. This document explains the flow so that implementers and maintainers can trace any datum from source to sink.

## End-to-end data flow summary

**Inputs to the system:** (1) Camera hardware (video device index, resolution, FPS from config). (2) Configuration files: `configs/app.yaml`, `configs/profiles/<name>.yaml`, and optionally `configs/calibration.yaml`. (3) User input: keyboard (scene switch, overlay toggle, quit) and the physical pointer (IR pen or bright spot) in the camera’s field of view.

**Outputs:** (1) Rendered frames on the display (Pygame window or fullscreen). (2) Pointer events delivered to the active scene (and optionally to WebSocket or uinput). (3) Optional overlay: FPS, threshold, pointer position. (4) On calibration path: written `configs/calibration.yaml`.

**Data flow in one frame (run loop):** Config is loaded once at startup. Each frame: camera yields a BGR image; the chosen detector (IR or bright) yields camera-space (x, y) or None; the tracker smooths and optionally clamps velocity; homography (if loaded) maps camera (x, y) to projector (x, y); the gesture layer turns position stream into PointerEvents (down/move/up); the Pygame adapter converts events to dicts; the active scene’s `on_pointer` and `update`/`draw` consume them; the overlay is drawn; the display is flipped. Below we detail each stage and the exact data in and out.

## 1. Configuration loading (before pipeline)

**Service:** Configuration is not a pipeline stage but a prerequisite. It runs at process start and when starting the app or calibration wizard.

**Functions involved:** `_load_app_config(config_dir)` (in `cli.py` and used by the app’s `_load_config`), and `load_calibration(path)` when running the main app.

**Data in:** `config_dir` (string, e.g. `"configs"`). File system: `config_dir/app.yaml` must exist (or defaults are used); `config_dir/profiles/<profile>.yaml` is optional; `config_dir/calibration.yaml` is optional for the main app.

**Data out:** A single merged dict for app config: `camera_index` (int), `resolution` (list of two ints: width, height), `detector` (string: `"ir"` or `"bright"`), `threshold` (int 0–255), `blur` (int, kernel size), and any keys from the selected profile (e.g. exposure hints). For calibration: a dict with `homography` (3×3 NumPy array), `bounds` (tuple of four ints), `resolution` (tuple of two ints), or `None` if file missing/invalid.

**Logic:** Load `app.yaml`; read `profile` name (default `"classroom"`); load `profiles/<profile>.yaml` if present and merge into app dict (profile keys override). Do not commit `calibration.yaml`; it is generated by the calibration wizard and read by the main app to get homography and resolution.

## 2. Capture (first pipeline stage)

**Service:** Acquire one frame from the camera in BGR format at the configured resolution and target FPS.

**Entry point:** `CameraCapture` in `smartpi.core.capture`. The app creates one instance per run; the calibration wizard creates one for the duration of the wizard.

**Data in:** At construction: `camera_index` (int), `width` (int), `height` (int), `fps` (float). These come from app config. No per-frame input.

**Data out:** `open()` returns `bool` (success). `read()` returns `(bool, np.ndarray | None)`: success flag and a BGR frame (shape H×W×3, dtype typically uint8) or `None` on failure. Resolution matches what was set (best effort by the driver).

**Logic:** OpenCV `VideoCapture(camera_index)`; set `CAP_PROP_FRAME_WIDTH`, `CAP_PROP_FRAME_HEIGHT`, `CAP_PROP_FPS`. Each `read()` calls `self._cap.read()`; if the device is not opened or read fails, return `(False, None)`. Release on exit or when the app/wizard exits.

**Order:** Capture is the first stage. The main loop and the calibration wizard both call `capture.read()` once per iteration. No other pipeline stage runs until a frame is available (or the app handles “no frame” by drawing a fallback and ticking at low rate).

## 3. Detect (second pipeline stage)

**Service:** From a single BGR frame, produce either one camera-space point (x, y) or None. Two detectors are available: IR blob (for IR pen) and bright point (for laser or bright LED).

**Entry points:** `detect_ir_blob(frame, threshold, blur_ksize, morph_ksize)` and `detect_bright_point(frame, threshold, blur_ksize)` in `smartpi.vision`. The app chooses one based on config `detector`.

**Data in:** `frame`: NumPy array BGR (H×W×3) or grayscale (H×W). `threshold`: int 0–255. `blur_ksize`: odd int or 0. For IR only: `morph_ksize`: odd int for morphology.

**Data out:** `tuple[float, float] | None`. If a pointer is detected: (x, y) in camera image coordinates (origin top-left, x right, y down). If none: `None`.

**Logic (IR):** Convert to grayscale and blur (`preprocess_grayscale`); threshold above `threshold` (`threshold_mask`); morphology close/open to clean the mask; find contours; take largest contour above area 10; compute centroid via moments; return (cx, cy) or None if no valid blob.

**Logic (bright):** Convert to grayscale and blur; find global max with `minMaxLoc`; if max value < threshold return None; else return (x, y) of max location.

**Order:** Detect runs immediately after a successful `capture.read()`. It is the only consumer of the raw frame in the main loop (aside from the calibration wizard, which also uses the frame for drawing targets). Output is the only input to the next stage (track).

## 4. Track (third pipeline stage)

**Service:** Temporal smoothing and optional velocity clamping of the 2D pointer position so that the cursor does not jump or jitter.

**Entry point:** `SmoothingTracker` in `smartpi.vision.track`. One instance per app run. The tracker is updated with camera-space points only in the calibration wizard; in the main app it is updated with **projector-space** points (after homography), so smoothing happens in display space.

**Data in:** `update(x, y)`: two floats (projector x, y in the main app; or camera x, y in wizard if used). Constructor: `alpha` (float 0–1, EMA factor), `max_velocity` (float, pixels per frame cap).

**Data out:** `update()` returns `(float, float)`: smoothed (x, y). On first call or after `reset()`, returns the input point. Internal state is the previous smoothed position.

**Logic:** If no previous position, store and return (x, y). Otherwise compute displacement (dx, dy); if `max_velocity > 0` and distance > max_velocity, scale (dx, dy) to that length; then EMA: `_x += alpha * (x - _x)`, same for y; return (`_x`, `_y`). When the detector returns None, the app calls `tracker.reset()` so the next detection is treated as fresh.

**Order:** In the main loop, after the detector returns, the app maps camera (x, y) to projector via homography (if calibration exists), then calls `tracker.update(proj_x, proj_y)`. So the order is: Detect → (optional Homography) → Track. Track output is the canonical “current pointer position” for the rest of the frame.

## 5. Calibration (one-time and per-frame use)

**Service:** (1) One-time: compute and save a 3×3 homography from four camera-point ↔ projector-point correspondences. (2) Per-frame: apply that homography to map camera (x, y) to projector (x, y).

**Entry points:** `solve_homography(src_pts, dst_pts)`, `save_calibration(path, homography, bounds, resolution)`, `load_calibration(path)`, `apply_homography(H, x, y)` in `smartpi.vision.calibrate`. Validation: `reprojection_error(calibration_path, src_pts, dst_pts)` in `smartpi.vision.validate`.

**Data in (solve):** `src_pts`: list of at least 4 (x, y) camera-space points. `dst_pts`: list of same length of (x, y) projector-space points in same order. Typically camera points come from the wizard (user taps); projector points are fixed (e.g. four corners in pixel space).

**Data out (solve):** 3×3 NumPy float array (homography H) or None if fewer than 4 points or RANSAC fails.

**Data in (apply):** H (3×3), x (float), y (float) in camera space.

**Data out (apply):** (px, py) in projector pixel coordinates.

**Logic (solve):** Convert lists to float32 arrays; call `cv2.findHomography(src, dst, cv2.RANSAC, 5.0)`; return H. Save/load: YAML with homography (list of 9 floats), bounds, resolution. Apply: `cv2.perspectiveTransform` with shape (1, 1, 2).

**Order:** Calibration wizard runs before the main app (user runs `smartpi calibrate` once). The main app calls `load_calibration()` at startup; every frame, after detection, it calls `apply_homography(calibration["homography"], cx, cy)` to get projector coordinates, then passes those to the tracker and event layer.

## 6. Interact (event generation and adapters)

**Service:** Turn the stream of (position or None) into a stream of PointerEvents (down, move, up) in projector space, and optionally feed those events to Pygame, WebSocket, or uinput.

**Entry points:** `emit_events_from_track(xy, prev_xy, prev_down)` in `smartpi.interaction.gesture`; `PointerEvent` and `PointerState` in `smartpi.interaction.events`; `pygame_event_from_pointer(ev)` and `push_to_pygame(ev, callback)` in `smartpi.interaction.adapters.pygame_adapter`.

**Data in (emit_events_from_track):** `xy`: current (x, y) in projector space or None (pointer lost). `prev_xy`: previous frame’s (x, y) or None. `prev_down`: bool, whether the pointer was considered “down” last frame.

**Data out:** Iterator of `PointerEvent` with attributes: x, y (float), state (PointerState: DOWN, MOVE, UP), timestamp (float), confidence (float). Events are emitted when: pointer appears (prev_xy None, not prev_down) → DOWN; pointer moves (xy and prev_xy differ) → MOVE; pointer disappears (xy None, prev_down) → UP.

**Logic:** If xy is None and prev_down, yield UP at prev_xy. If xy is not None and prev_xy is None and not prev_down, yield DOWN at xy. If xy and prev_xy both set and (x,y) changed, yield MOVE at xy. The app maintains prev_xy and prev_down between frames and passes them into `emit_events_from_track`; it then forwards each event to the scene via `pygame_event_from_pointer(ev)` → dict with type, x, y, state, timestamp.

**Order:** After the tracker update, the app has current pointer (or None) and prev_xy/prev_down. It calls `emit_events_from_track(pointer, prev_xy, prev_down)` and for each yielded event calls `scene.on_pointer(pygame_event_from_pointer(ev))`. Then it sets prev_xy = pointer and prev_down = (pointer is not None). So interact runs after track and before draw.

## 7. Render (scenes and overlay)

**Service:** Draw the current scene (or scene selector message) and optionally an overlay (FPS, threshold, pointer position). Scenes react to pointer events and animate (e.g. particles).

**Entry points:** `run_app(fullscreen, config_dir)` in `smartpi.renderer.pygame_app` (main loop); `SandboxDrawScene`, `ParticleFieldScene`, `StemWidgetsScene` with methods `on_pointer(ev)`, `update()` (optional), `draw(surface)`; `draw_overlay(surface, fps, pointer_xy, threshold)` in `smartpi.ui.overlay`.

**Data in (scene):** `on_pointer`: dict with "type", "x", "y", "state", "timestamp". `draw`: Pygame Surface (screen). `update`: no arguments (uses internal state).

**Data out:** No return values; scenes mutate internal state (strokes, particles) and draw to the surface.

**Logic (main loop):** Handle Pygame events (QUIT, KEYDOWN for 1/2/3/O/Esc). Read frame; if no frame, draw fallback and overlay, tick 10 FPS. Else detect → homography → track → emit_events_from_track → on_pointer for each event. Update scene if has `update`; call scene.draw(screen); call draw_overlay(screen, fps, pointer, threshold); flip display; tick clock 30. So render runs after interact and is the last consumer of pointer state for visualization.

**Order:** Render is the final pipeline stage each frame. Operate (below) is the outer shell: CLI, healthcheck, diagnostics, calibration entry points.

## 8. Operate (CLI and scripts)

**Service:** Entry points for run, calibrate, healthcheck, export-diagnostics. Not per-frame; they start the pipeline or perform one-off checks.

**Data in/out:** CLI: `smartpi run` loads config, calls `run_app`; `smartpi calibrate` calls `run_wizard`; `smartpi healthcheck` opens camera, measures FPS, prints PASS/FAIL and optional CPU temp; `smartpi export-diagnostics` calls `bundle_diagnostics` and writes a tarball. See the CLI and operations deep-dive document for full function-level I/O.

**Order:** Operate is the outermost layer. User invokes one command; that command invokes one or more of: run_app, run_wizard, healthcheck logic, or bundle_diagnostics. No data flows between commands except via files (config, calibration).

## Summary table: data at each boundary

| Stage        | Input data                          | Output data                          |
|-------------|-------------------------------------|--------------------------------------|
| Config      | config_dir, YAML files              | Merged app dict; calibration dict   |
| Capture     | camera_index, w, h, fps             | (success, BGR frame)                 |
| Detect      | frame, threshold, blur, (morph)     | (x, y) or None (camera)             |
| Homography  | H, cx, cy                           | (px, py) (projector)                 |
| Track       | px, py (or cx, cy in wizard)        | (smooth_x, smooth_y)                |
| Interact    | xy, prev_xy, prev_down              | PointerEvent(s)                      |
| Adapter     | PointerEvent                        | dict for scene or socket/uinput     |
| Scene       | event dict, surface                 | (state change, pixels on surface)    |
| Overlay     | surface, fps, pointer_xy, threshold| (pixels on surface)                  |
